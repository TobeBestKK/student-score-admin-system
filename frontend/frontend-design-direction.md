# 前端设计方向说明书

## 1. 设计目标

### 1.1 文档目的

本说明书用于统一 `student-score-admin-system` 前端的视觉方向、页面气质、组件使用方式和图表表达规则，避免后续开发中出现页面风格漂移、组件混用失控、颜色体系失衡的问题。

### 1.2 适用范围

本说明书适用于整个前端系统，包括但不限于以下页面和模块：

- 登录页与门户类页面
- 后台工作台与概览页
- 列表页与检索页
- 表单页与详情页
- 数据分析页与图表页
- 空状态、异常状态、反馈状态页面

### 1.3 设计目标

- 建立稳定、正式、克制的“现代校务门户”视觉基调
- 让学生端与教师端在同一系统内拥有统一但可区分的交互体验
- 让设计 token 可以直接指导 `Tailwind`、`shadcn-vue`、`Element Plus` 和 `ECharts` 的落地
- 优先服务信息阅读、数据扫描、表单填写和后台操作效率，而不是视觉炫技

## 2. 视觉基调与系统语气

### 2.1 核心气质

系统整体气质定义为：

- 正式
- 清晰
- 可信
- 有秩序
- 低干扰

它应该像“高校教务门户”和“教学数据系统”的结合体，而不是营销网站、数据大屏或互联网活动页。

### 2.2 视觉关键词

- 浅色背景
- 青蓝绿色主色
- 低饱和中性色
- 细线条与规则网格
- 轻量校园轮廓或成绩曲线背景
- 小范围暖色强调

### 2.3 文案语气

- 用词正式、准确、简洁
- 避免口语化、营销化、情绪化表达
- 优先使用业务语义，如“成绩分析”“教学评估”“学业概览”“异常预警”
- 操作文案直接，不使用夸张表达

### 2.4 背景表达原则

- 背景只能提供系统语境，不得抢占主体注意力
- 校园剪影、网格、曲线、淡色形状可用于登录页、门户页和空状态页
- 业务页背景应弱化，优先使用纯色底和轻边界容器
- 不使用深色海报式大背景作为全站主方向

## 4. 组件与交互规则

### 4.1 技术栈职责边界

- `shadcn-vue`
  - 用于登录页、卡片、轻表单、标签页、提示、按钮、轻量面板
  - 适合建立统一视觉基线
- `Element Plus`
  - 用于后台复杂业务控件
  - 例如复杂表格、分页、日期选择、树、穿梭框、弹层业务组件
- `Tailwind v4`
  - 用于页面布局、间距、颜色 token 消费、局部状态表达
  - 不应继续直接散落大量临时十六进制颜色
- `ECharts`
  - 用于所有图表和数据可视化
  - 图表颜色必须优先使用文档中的图表色板

### 4.2 通用组件规则

- 卡片默认圆角使用 `radius-md`
- 输入框、按钮、选择器高度优先使用 `40px`
- 主要按钮使用 `brand-primary`
- 次级按钮优先使用白底描边，而不是大面积灰色按钮
- Tabs、Segmented 控件要简洁，避免过厚边框和高饱和底色
- 图标只做语义补充，不作为页面主视觉主体

### 4.3 表单规则

- 标签在上、输入在下，保证扫描路径稳定
- 校验错误在字段下方展示，使用 `state-danger`
- 占位文案只做提示，不承担说明性文字
- 长说明应放在字段上方或字段分组说明中

### 4.4 反馈规则

- 成功提示使用浅绿色背景 + 深绿色文字
- 警告提示使用浅黄色背景 + 深黄色文字
- 错误提示使用浅红色背景 + 深红色文字
- 不使用浮夸动画；反馈以出现、淡入、位置稳定为主

## 5. 页面模板规范

### 5.1 登录 / 门户页

- 使用浅色全屏背景
- 背景允许少量网格、淡曲线、校园剪影
- 主体为居中或偏中轴登录面板
- 不叠加复杂数据卡片和厚重海报结构

### 5.2 后台内容页

- 结构以“页标题 + 筛选区 + 内容区”为主
- 内容区优先使用表格、分区块和稳定栅格
- 避免把整页包成大卡片

### 5.3 列表页

- 重点突出筛选、检索、批量操作和表格主体
- 顶部信息密度要高，但层级必须清楚
- 不在列表页使用装饰性大背景

### 5.4 表单页

- 大表单优先分组展示
- 每组只承载一类业务信息
- 操作按钮区位置固定，避免每页飘忽不定

### 5.5 数据分析页

- 页面应先给结论层，再给趋势层，最后给明细层
- 图表与数字卡片要可比较、可扫描
- 颜色应克制，避免每个图都用不同高饱和色

### 5.6 空状态 / 错误状态页

- 使用简短说明和明确操作
- 可使用轻量插图或图标，但不得喧宾夺主
- 语气保持正式，不使用娱乐化文案

## 6. 图表与数据表达

### 6.1 图表原则

- 图表服务决策，不服务装饰
- 优先让横纵坐标、图例、趋势、异常点清楚可读
- 控制图表种类，不为“丰富”而堆叠复杂图表

### 6.2 配色规则

- 主图表使用 `chart-1` 到 `chart-5`
- 同一页面内图表色应保持稳定映射
- 成绩相关的主趋势优先用 `chart-1`
- 风险、异常、预警使用 `state-warning` 或 `state-danger`

### 6.3 图表留白规则

- 图表外围保留足够边距，避免标题、图例、坐标拥挤
- 卡片内图表与标题、筛选区之间至少保持 `space-4`
- 不使用厚重阴影和强装饰边框包裹图表

## 7. 禁忌项

- 不使用深色海报式全页背景作为默认风格
- 不使用高饱和渐变作为主视觉基底
- 不使用过度玻璃拟态、强反光、重模糊
- 不使用营销式大 Hero、宣传文案式首页结构
- 不在后台业务页加入无意义装饰卡片
- 不混合多套视觉体系导致 `shadcn-vue` 与 `Element Plus` 风格冲突
- 不在页面中继续直接硬编码大量零散颜色值
- 不把图表做成“展示墙”，牺牲可读性

## 8. 开发落地规则

### 8.1 样式实现原则

- 新页面开发前先查 token，再写样式
- 优先使用语义化颜色和尺寸，而不是临时拍脑袋选值
- 后续应逐步把常用颜色和尺寸抽象为统一 CSS 变量或 Tailwind 语义层

### 8.2 组件使用原则

- 轻量界面优先用 `shadcn-vue`
- 复杂业务控件优先用 `Element Plus`
- 如果 `Element Plus` 控件进入页面，外层布局和页面结构仍应遵守本说明书

### 8.3 代码审查检查项

提交前至少检查以下内容：

- 是否使用了文档规定的主色、背景色和文本层级
- 是否存在不必要的大圆角、大阴影、重背景
- 是否出现多套按钮风格并存
- 是否存在高饱和无语义颜色
- 图表颜色是否遵守图表色板
- 页面是否符合对应模板类型，而不是任意拼装

### 8.4 后续演进建议

- 第一阶段：先以本说明书约束新页面
- 第二阶段：逐步把现有页面颜色、圆角、阴影、按钮统一到 token
- 第三阶段：形成共享主题层，减少页面级硬编码样式

---

本说明书是当前项目的前端视觉基线。后续若出现新的业务场景，应优先在本说明书中补充规则，再进入实现阶段。
